What this is

The Retail Benchmarks dashboard tracks sales per square foot for tenants at shopping centers JRB is evaluating for physical retail. It's how we score locations and compare JRB's expected PSF to beauty and activewear comps.

Every row is one tenant at one center: {retailer, location, city, state, annual_sales, size_sf, sales_psf, category}. The data lives in Supabase and updates show up immediately on the live URL.

Where data comes from

  • Landlord disclosures — emailed PDFs, broker decks, leasing one-pagers
  • Broker conversations — directly transcribed numbers
  • Public filings & press — quarterly earnings, retail analyst notes
Don't use: guesses, scraped retail data without source, or numbers from competitors' press releases without a credible second source. If you can't say where a number came from, don't enter it.

Flow A — Adding a new comp ADD

Use this when a brand-new tenant or a new shopping center comes into the dataset.

  1. Open jrb-retail-benchmarks.vercel.app.
  2. Click the black + Add Tenant button in the top-right.
  3. Fill in the modal. Required fields: Retailer, Location, City, State, Category. Optional: Sales PSF, Annual Sales, Store Size.
  4. If the retailer or location already exists in the data, start typing — the autocomplete will show matches. Picking an existing retailer auto-fills its Category. Picking an existing location auto-fills City and State.
  5. Click Save Tenant. The page refreshes and the new row appears in the table.

Flow B — Editing an existing comp EDIT

Use this when new numbers come in for a tenant we already track (e.g., updated annual sales for a new year).

  1. Find the tenant in the All Tenants table. Filter by city, retailer, or category if needed.
  2. Hover the row — a pencil icon appears on the right side.
  3. Click the pencil. The modal opens pre-filled with the current values, titled "Edit Tenant".
  4. Change only the fields that need updating. Leave the rest alone.
  5. Click Save Changes. The row updates in place.
Important: Don't use + Add Tenant to "update" an existing row — that creates a duplicate. Always edit via the pencil.

Flow C — Deleting a bad row DELETE

Use this when a row is wrong, duplicated, or the source turned out to be unreliable.

  1. Find the row in the table.
  2. Hover the row — a trash icon appears next to the pencil.
  3. Click the trash. A confirmation popup asks you to confirm.
  4. Click OK. The row is gone from Supabase. (This cannot be undone — if you delete by mistake, re-add the row from Flow A.)

Flow D — Bulk update from a PDF, CSV, or screenshot BULK

Use this when you have a stack of new numbers — a landlord disclosure PDF, an exported spreadsheet, or a screenshot of a table from a deck. Instead of typing each row, hand the file to Claude.

Inside Claude (Cowork or Claude Code)

  1. Drag the file (PDF, CSV, or screenshot) into the chat. Or paste a path if you have one.
  2. Type /update-comps and reference the file. Example: /update-comps the disclosure I just dropped.
  3. Claude parses the file, fetches existing data from Supabase, and shows you a diff:
    • NEW — rows that don't exist yet (will be added)
    • UPDATE — rows that already exist with different numbers (only changed fields shown)
    • SKIP — rows that match exactly, no change needed
    • AMBIGUOUS — Claude can't tell which existing row to match (you'll be asked)
  4. Review the diff carefully. If anything looks wrong, type edit and tell Claude what to fix.
  5. When the diff is right, type yes. Claude writes the changes to Supabase.
  6. Refresh the dashboard to verify the new numbers are live.
Source tips: PDFs work best when they're text-based (not scanned images). For screenshots, frame the table tightly and make sure the retailer + center name are both readable. For CSVs, include a header row with column names so Claude can map fields.

Sanity checks before saving

A few quick gut-checks to catch bad numbers before they go in:

  • PSF range: Most comps land between $400 and $5,000 PSF. Above $7,000 means luxury/jewelry/Apple — make sure that fits the retailer.
  • Sales ÷ size = PSF: If all three are given, the math should match within ~5%. If it doesn't, one of them is wrong.
  • Annual sales: Almost always between $1M and $50M for specialty retail. Below $500K = probably wrong unit (was it monthly?). Above $100M = probably the entire chain, not one store.
  • Store size: Beauty is usually 1,000–4,000 SF. Apparel is 2,000–10,000 SF. Department stores are 30,000+. Restaurants vary.
  • Category: Pick from the existing list. If a retailer doesn't fit any (e.g., a museum gift shop), ping Cody before adding a new category.

Troubleshooting

"Save failed: HTTP 401"

The Supabase row-level security policies aren't set up for this action. Ping Cody — he'll add the missing policy.

The row doesn't update / disappear after saving

Hard-refresh the page (⌘ + Shift + R). If it still doesn't show, the write might have silently failed. Open the browser dev tools (⌘ + Option + I), check the Console tab for red errors, and screenshot to Cody.

I edited the wrong row

If you remember the original values, edit again and put them back. If not, ping Cody — Supabase keeps point-in-time recovery for the recent past.

The PDF parser missed rows

It happens with messy layouts. Either (a) crop a screenshot of just the missed rows and run /update-comps on that, or (b) add the missing rows manually via Flow A.

When to ping Cody: a) the dashboard is fully broken (empty or 500 errors), b) you need a new category added, c) you're not sure if a number is real, d) Save fails with 401/403, e) you accidentally deleted something important.
