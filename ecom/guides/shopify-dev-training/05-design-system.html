Stage 03 — Build

One design system. Loaded before any code is written.

The biggest leverage in our build pipeline isn't the AI — it's the rule that the design system loads before the AI starts writing. Every page begins brand-accurate. Nothing has to be corrected later.

The mandatory pre-build sequence

Before any HTML, CSS, or Liquid gets written for JRB, three files load in order:

1. ~/.claude/skills/shared/component-templates.md // 51 paste-ready section templates 2. /jrb-design-system // Tokens, fonts, colors, button standards 3. /jrb-section-library // 108 existing sections to check before building new

Skipping any of those is a compliance failure. The QA skill catches it later, but the loss is wasted output.

Tokens

The whole design system fits on one screen.

  • Colors. #000 black, #fff white, #f8f8f8 light gray, #f6f2eb warm cream, #ff2e92 hot pink. Section backgrounds alternate #fff and #f8f8f8. Cream is utility-bar and footer only.
  • Headlines. Canela Light, sentence case. Never Geogrotesque for large headlines.
  • Body. RingsideWide. Always #000. Never #555, #666, #333, or --color-muted.
  • Subheadlines. ArticulatCF Bold, ALL CAPS, letter-spacing 0.3px. Never substitute with Geogrotesque.
  • Mono. BaseMonoWide for labels and data.
  • Buttons. Always the .button class. Default black background. Pink only when explicitly requested.
  • Corners. 2px. Sharp, not pill.

The skill

/jrb-design-system
The executable reference
Loads tokens, patterns, and the JRB rules into context. Every page-builder skill calls this first. The rules aren't in a PDF — they're loaded directly into the model, which means they shape the output, not just review it.
Allowed tools: Read, WebFetch

The component templates file

Fifty-one paste-ready HTML+CSS templates for the section types we build over and over — heroes, CTA bands, review carousels, press logos, FAQ, founders, steps, product cards, trust bars. Lives at:

~/.claude/skills/shared/component-templates.md // The single source of truth for reusable component HTML+CSS. // Loaded BEFORE the design system reference.

The "never write CSS from scratch" rule

This is the rule that beats every other rule. Before writing any new CSS for any UI element, check four sources in order:

  • 1. component-library.html in the current project
  • 2. Existing sections in the project that already implement the element
  • 3. ~/.claude/skills/shared/component-templates.md
  • 4. The JRB theme — buybox, existing sections, __index-css.liquid

If the element exists anywhere in those four sources, copy the exact pattern verbatim. Don't approximate, simplify, or resize. This applies to swatches, buttons, stars, trust badges, cards, accordions, and every other reusable element.

Why this matters: every drift becomes a future bug. A button that's "almost" the JRB button breaks the visual consistency of the site one place at a time. The cumulative effect is a brand that looks slightly off everywhere. Copying verbatim prevents that.

What's NOT in the design system

  • Auto-advance carousels. Manual click and swipe only. No setInterval.
  • Emojis. Anywhere.
  • Orphan words. Use text-wrap: balance or hide overflow on mobile.
  • Scroll animations, unless explicitly requested.
  • The colors #1A1A1A, #FAF8F5, #C73A3A. Common AI substitutions. Caught by QA.

The shared reference files

Three files in ~/.claude/skills/shared/ are loaded into every JRB build session. They live alongside the skills (private repo) but are worth knowing by name:

  • component-templates.md — 51 paste-ready section templates (heroes, CTAs, reviews, press, FAQ, founders, steps, product cards, trust bars)
  • jrb-approved-patterns.md — patterns that have been validated in production and are safe to reuse
  • shopify-section-safety-rules.md — Liquid scoping, JS isolation, schema conventions for any HTML-to-Liquid conversion
