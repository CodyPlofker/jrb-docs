Foundations — 01

The four-stage pipeline.

Most eCom teams treat customer research, CRO, design, and dev as four separate functions with four separate tools. Here they're one workflow. Each stage feeds the next, and Claude Code does the connective tissue.

Loom video coming soon

The shape of the system

Four stages. Each one is a module in this hub.

  • Intelligence. Five customer-data sources merged into one shared file. Every downstream skill reads from it.
  • CRO. Test ideas scored with ICE, prioritized, sized to traffic, and run through Intelligems. Results loop back into the roadmap.
  • Build. Brand-accurate sections and pages from a single design system. The 108-section library means we rarely build from scratch.
  • Ship. HTML to Liquid, three QA gates, then a PR to the develop branch on the theme repo. Never main.

Why this works

Three things change when the four stages share a system instead of bouncing between tools.

  • Every test is grounded in customer language pulled from real interviews and reviews — not gut feel.
  • Every page matches the brand on first render because the design system is loaded automatically before any code is written.
  • Every skill calibrates on results. When a test wins, ICE scores update. When a page underperforms, Page Watchlist alerts Slack.

Where the leverage is

The leverage isn't in any single skill. It's in the fact that /jrb-cro reads the same file /newlandingpage reads, which is the same file /optimize reads. One source of truth, many specialists.

The shared customer file lives in a private GitHub repo. Every skill that touches it does an automatic git pull before reading and a git push after writing. The whole team stays in sync without thinking about it.

How to use this hub

Read the modules in order. Each one is short. Each one ends with a "next" link to the following module. The total time to read the hub end-to-end is under an hour.

If you want a deeper read on any stage, every module links to the actual skill files on GitHub.
