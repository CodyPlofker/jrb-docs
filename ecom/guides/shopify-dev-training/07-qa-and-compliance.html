Stage 03 — Build

Three gates between a build and the live store.

Even with the design system loaded, things drift. Three QA skills run before any code reaches the theme repo. Each one checks a different layer — brand, repo safety, and section conventions.

Gate 1 — Brand compliance

/jrb-qa
Post-build brand checker
Reads generated Liquid, HTML, or CSS and validates against the JRB design system. Outputs a pass/fail report with line numbers and suggested fixes. Catches font substitutions, wrong color values, missing button classes, broken naming conventions.
Run after every build. Before every PR.

Common things /jrb-qa catches:

  • Body text colored #666 or #555 instead of #000
  • Headlines using Geogrotesque instead of Canela
  • A "new" FAQ section with custom JS toggles instead of reusing casting-faq.liquid
  • Placeholder.com images that survived from a draft
  • Section files using the wrong naming convention (missing ccig- or cclp- prefix)
  • Auto-advance carousels with setInterval

Gate 2 — Repo safety

/ee-audit
Pre-PR safety against EcomExperts rules
Before opening a PR against the theme repo, this skill checks every changed file against the EcomExperts safety rules — no generated files, no committed build artifacts, no broken Liquid syntax, no edits to files outside the change scope.

The EcomExperts repo has a "no generated files" rule. Built artifacts that look like source files (compiled CSS, generated Liquid) are not allowed in commits. This skill catches them before the PR is open and the team has to ask for a force-push.

Gate 3 — Section conventions

/section-audit
Section-level pattern check
Audits a single Shopify section file against the JRB design system and existing theme patterns. Different from /jrb-qa in that it focuses on section structure — schema settings, block organization, scoped CSS, the section-level rules that aren't visible in a rendered preview.

The order they run

For a section build:

1. /jrb-qa // Brand compliance on the rendered output 2. /section-audit // Section structure and Liquid conventions 3. /ee-audit // Repo safety, last check before PR

For an LP that lives in jrb-landing-pages rather than the theme:

1. /jrb-qa // Brand compliance only // (no theme PR, so no /ee-audit needed)

Why three gates instead of one

Each gate catches a different category of failure:

  • Brand drift is visible — wrong color, wrong font. Caught by /jrb-qa.
  • Convention drift is structural — wrong file name, missing schema, broken block hierarchy. Caught by /section-audit.
  • Repo drift is procedural — wrong files committed, generated artifacts, edits outside scope. Caught by /ee-audit.

Combining them into one skill loses precision. Keeping them separate means each one has a tight, well-tuned ruleset.

Try it
After building any section, run /jrb-qa, then /section-audit, then /ee-audit in order. Fix every Critical and Warning before opening the PR. The skills are fast — the whole pass is under two minutes for a typical section.
