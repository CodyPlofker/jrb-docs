One mockup, many sections.

The build skills produce HTML mockups. Shopify needs Liquid sections with scoped CSS and editable schema settings. This skill bridges them in one command.

The skill

/html-to-shopify
Convert HTML into multiple Liquid sections
Takes a complete HTML mockup and splits it into separate .liquid files, one per section. Scopes the CSS so sections don't bleed into each other. Extracts editable text, images, and links into Shopify schema settings so merchants can edit them in the admin without touching code.

Naming conventions

Section files follow strict prefixes so the theme's section list stays organized.

  • ccig-*.liquid for general sections (e.g. ccig-hero-banner.liquid)
  • cclp-*.liquid for landing-page-specific sections (e.g. cclp-miracle-balm-hero.liquid)
  • Theme-native sections (existing) keep their original names — never rename existing files

What the conversion does

Input: one HTML file with seven sections mockup.html Output: seven Liquid files, each with scoped CSS and schema sections/ ├── cclp-miracle-balm-hero.liquid ├── cclp-miracle-balm-problem.liquid ├── cclp-miracle-balm-proof.liquid ├── cclp-miracle-balm-ingredients.liquid ├── cclp-miracle-balm-shades.liquid ├── cclp-miracle-balm-faq.liquid └── cclp-miracle-balm-cta.liquid

Schema extraction — what becomes editable

Anything that should be editable in the Shopify admin gets pulled into the schema. Headlines, body copy, button text, button URLs, image fields, and image alt text. Fixed brand patterns — fonts, colors, spacing, button styles — stay in the CSS.

{% schema %} { "name": "Miracle Balm Hero", "settings": [ { "type": "text", "id": "headline", "label": "Headline", "default": "A new gold standard." }, { "type": "image_picker", "id": "hero_image", "label": "Hero image" } ] } {% endschema %}

The pre-build safety rules

Lives at ~/.claude/skills/shared/shopify-section-safety-rules.md. Loaded automatically before any HTML-to-Liquid conversion. Covers:

  • CSS scoping — no global selectors that escape the section
  • JavaScript scoping — IIFE pattern, no globals, no document.querySelector without a section root
  • Liquid syntax — proper {% schema %} JSON, valid block hierarchies, translation keys where required
  • Image handling — Shopify CDN URLs only, no Air imgix URLs in Liquid
  • Performance — lazy-load images below the fold, no oversized assets

If the resulting Liquid files break theme conventions or violate the safety rules, /jrb-qa + /section-audit catch it before the PR is opened. Don't skip QA after a conversion just because the build skill produced clean HTML.

SPA architecture note

The JRB theme is a SPA. JavaScript and CSS that need to work across navigation must live in the /app folder, not in section-local {% javascript %} tags. Liquid-only changes are safe to put in section files. JS-driven behavior gets routed to /app by the build pipeline.

Try it
Build a section as HTML with /jrb-section-builder. Then run /html-to-shopify on the output. Compare the input HTML to the output Liquid — note what got pulled into schema and what stayed in CSS.
