Stage 04 — Ship

Branch from develop. PR to develop. Never main.

The theme push workflow is the most rule-bound part of the system. It has to be — the theme is production. One bad merge takes down a section. The skill encodes every safety check so the workflow can run without thinking.

The full GitHub PR workflow
Pre-push QA, dev-theme preview, branch creation, scoped commits, and a PR opened against the develop branch with the right reviewers and labels. Ten steps, all encoded in the skill so nothing gets skipped.

The non-negotiables

  • PRs target develop. Never main. main is only updated by the EcomExperts deploy pipeline.
  • Section prefixes are cclp- or ccig-. Anything Claude-built carries the prefix. Files without the prefix are theme-native and stay untouched.
  • Never push the whole theme. Always scoped pushes — only the specific files in the change.
  • Never push to the published theme. Always ask which preview theme to push to. The skill enforces this with a prompt before any push runs.
  • No generated files in commits. EcomExperts repo rule. /ee-audit catches violations before the PR opens.

The branch naming convention

cclp/sleepless-nights-kit // new landing page cclp/fix-hero-mobile-aspect // bug fix cclp/refactor-shared-styles // refactor

The full sequence

  1. Run /ee-audit and fix all Critical and High issues
  2. Run /simplify and address actionable findings
  3. Update the section registry in jrb-theme-docs/claude-created-sections.json
  4. Ask the user which preview theme to push to (never assume)
  5. Scoped shopify theme push with --only flags
  6. Share the preview URL inline so it can be verified
  7. git checkout develop && git pull to sync
  8. Branch with cclp/[descriptive-name]
  9. Stage and commit only the changed files
  10. Push and open a PR against develop using gh pr create

Why this is structured this way

Three reasons.

  • Scoped pushes prevent collateral damage. shopify theme push without --only would overwrite every file in the preview theme. Scoping limits the blast radius to the actual change.
  • The dev-theme preview catches what local testing misses. Liquid syntax is fine locally, but Shopify renders it with real product data, real cart context, and real customer login state. A section that works on a static HTML preview can break on a live store.
  • The PR is the conversation. EcomExperts engineers review every PR before it merges to develop. The PR description, the screenshots, and the preview URL are how the change gets explained.

The dev preview URL gets shared inline after every push. It's a saved feedback memory — Cody asked for this so he can verify on his phone without scrolling back through the terminal output. The skill always includes the URL.

The SPA architecture rule

JS and CSS that need to work across navigation must live in the /app folder, not in section-local {% javascript %} tags. Liquid-only changes are safe to put in section files. The skill catches misplaced JS during the audit pass.
