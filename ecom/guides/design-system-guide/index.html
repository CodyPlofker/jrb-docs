Practical Guide

How to build a design system for Claude Code

A step-by-step guide to creating a design system file that gives AI perfect brand consistency on every page it builds. Uses Jones Road Beauty's production system as a worked example.

01

Why a design system matters for AI

Without a design system, every AI-generated page is a coin flip. Claude will make reasonable choices, but they'll be different reasonable choices each time.

A design system for Claude Code is a single Markdown file that Claude reads before building anything. It defines your fonts, colors, spacing, components, and rules. Once it exists, every page Claude builds automatically follows the same visual language.

Three differences from a traditional style guide

1. Machine-readable. Written in Markdown with CSS code snippets Claude can copy directly into the pages it builds. Not a PDF or Figma file — a text file that lives in your project.

2. Prescriptive, not descriptive. Says ALWAYS use #000 for body text, NEVER #555 instead of "body text should be dark." The explicit prohibitions are what catch mistakes.

3. Living document. Updated every time you notice a pattern. Push to GitHub, and every future build benefits immediately. The system gets smarter over time.

Real example: JRB's design philosophy

Here's the opening of Jones Road Beauty's production design system — the first thing Claude reads:

## Design Philosophy

The JRB design system embraces a clean, premium beauty brand aesthetic with:
- High readability and generous whitespace
- Minimal decoration, content-first layouts
- Premium but accessible (not luxury-distant)
- Wide monospace (BaseMonoWideWeb) for buttons and utility text
- Condensed bold sans (GeogrotesqueCondBoldWeb) for small display labels
  — NOT for large page/section headlines
- Body text in RingsideWideWeb, a refined sans-serif
- Canela serif for section titles and editorial moments
- Sharp 2-3px corners on buttons and cards (never 0, never rounded)
- Warm neutral palette anchored by #f6f2eb backgrounds and #000 text
- Brand accent: #ff2e92 (hot pink/magenta)

This paragraph sets the tone for everything. Claude uses it to make judgment calls on details the rest of the file doesn't explicitly cover.

02

Anatomy of a design system file

A complete design system file has 8–10 sections. Here's what each covers and why it matters. You don't need all of these on day one — start with typography and colors, then expand.

Section What it covers Why it matters
Design Philosophy Overall aesthetic direction in plain language Sets the tone. Claude uses this to make judgment calls on things the rest of the file doesn't explicitly cover.
CSS Methodology BEM naming, specificity rules, nesting, property order Prevents inconsistent code style across pages. Claude writes cleaner CSS when the rules are explicit.
Typography Font families, sizes, weights, usage rules, @font-face The #1 source of brand inconsistency without rules. Gets the most detail in any good design system.
Colors Hex values, CSS variables, semantic colors, wrong values to avoid Prevents "close but wrong" color drift. The explicit Don'ts are as important as the Do's.
Spacing Token scale (4px–96px), container max-widths, section padding Creates visual rhythm and consistency across pages. Prevents arbitrary padding/margin choices.
Components Buttons, cards, accordions, forms — with full CSS Reusable patterns Claude can copy directly. The most impactful for productivity.
Breakpoints Mobile-first media queries, responsive behavior Consistent responsive behavior. Without this, Claude guesses at breakpoint values.
Imagery Aspect ratios for product, editorial, hero, UGC photos Prevents wrong crops — landscape ratios for portrait photos is a common mistake.
Do's and Don'ts Explicit ALWAYS/NEVER rules with examples The highest-impact section. Catches the most mistakes. This is where you encode hard-won lessons.

JRB's file is 1,586 lines. That didn't happen on day one. It started small and grew through the feedback loop described in section 9. Your V1 can be 100 lines.

03

Deep dive: Typography

Fonts carry more brand identity than colors. Using the wrong font in the wrong place is the most common form of brand drift — and the design system prevents it.

What to document

For each font family, you need: the CSS font-family value (with fallbacks), the weight, what it's used for, the size range, any text-transform rules, and what it should NOT be used for.

Real example: JRB's font hierarchy table

JRB uses 5 font families, each with a specific job. This table is the single most-referenced part of their design system:

| Role                        | Font                    | Example                              |
|-----------------------------|-------------------------|--------------------------------------|
| Large headlines (hero, titles)  | Canela Light 300          | "The 10-second complexion upgrade"   |
| Section eyebrows / LP labels  | Geogrotesque Bold 18px    | "THE PROBLEM" - "HOW IT WORKS"       |
| Small display labels           | Geogrotesque Bold, upper  | "INSTANT POLISH"                     |
| Buttons, tabs                  | BaseMono Wide 11-13px     | "SHOP NOW"                           |
| Trust/value text (below CTA)   | RingsideWide 14px         | "Free shipping on $50+"             |
| Body text, descriptions        | RingsideWide Book         | "One swipe delivers a natural glow." |
| Editorial moments              | Canela Light Italic       | "It's the only makeup I actually use." |

The NEVER pattern

Documenting what a font should not be used for is as important as what it should be used for. Without these rules, Claude will sometimes use the wrong font in a visually plausible but off-brand way.

Weak rule

"Geogrotesque is for display text and labels."

Claude might use it for a large hero headline — it's technically "display text."

Strong rule

"Geogrotesque: Use for small display labels (18px), product names, accordion triggers. NEVER for large page/section headlines — those use Canela."

Rule of thumb: Max 3 font families per page (excluding script/accent fonts). If you find yourself needing more, the hierarchy isn't clear enough.

04

Deep dive: Colors

Fewer colors, documented precisely. The explicit list of "wrong" colors is what makes this work — Claude sometimes generates visually similar but incorrect values.

Real example: JRB's 3-color system

JRB uses just three primary colors. Everything else is a neutral:

Black
#000
Warm Cream
#f6f2eb
Hot Pink
#ff2e92
--primary-color: #ff2e92;  /* Brand pink — CTAs, accents, sale badges */
color: #000;               /* Primary text, borders, button backgrounds */
background: #f6f2eb;      /* Warm off-white — signature JRB background */

Document the wrong values

This is the single most impactful thing you can do in your color section. Without explicit prohibitions, Claude will occasionally generate "close" values that look almost right but aren't your brand:

Don't

#1A1A1A for black — use #000

#FAF8F5 for background — use #f6f2eb

#C73A3A for accent — use #ff2e92

Do

List every hex value you've seen Claude generate incorrectly. Each one you add prevents that mistake from ever happening again.

05

Deep dive: Components

Components should include copy-pasteable CSS, not just descriptions. Claude works best when it can use your exact code.

Start with your most-used component — usually a button. Document the full CSS declaration, hover states, variants, and common mistakes. This gives Claude a template for how detailed other components should be.

Real example: JRB's button component 
.button {
  background: #000;
  border: 1px solid #000;
  border-radius: 2px;
  color: #fff;
  cursor: pointer;
  display: inline-block;
  font-family: BaseMonoWideWeb, Courier, 'Courier New', sans-serif;
  font-size: 13px;
  font-weight: 550;
  height: 40px;
  letter-spacing: .4px;
  line-height: 40px;
  padding: 0 36px;
  text-align: center;
  text-transform: uppercase;
  white-space: nowrap;
  transition: all .4s cubic-bezier(.39, .575, .565, 1);
}

/* Hover only on devices with a mouse */
.has-mouse .button:not(:disabled):hover {
  background: #fff;
  border-color: #d0d0d0;
  color: #000;
}

/* Variants */
.button--white   { background: #fff; border-color: #fff; color: #000; }
.button--outline  { background: transparent; color: #000; opacity: .6; }
.button--pink    { background: var(--primary-color); color: #fff; }

Every property is specified. Claude never has to guess font-size, font-weight, or border-radius — it copies from the design system directly.

Key principle: Document the CSS, not a description of the CSS. "Buttons should be dark with a monospace font" is less useful than the actual CSS declaration block.

06

File structure — where to put it

The design system lives as a "skill" in Claude Code's skills directory. Claude reads it automatically when the skill is invoked.

~/.claude/skills/
  your-design-system/
    SKILL.md              # The design system itself
    references/
      css-standards.md    # Deeper CSS methodology (optional)
      component-lib.md    # Extended component catalog (optional)

The frontmatter

Every skill file starts with YAML frontmatter that tells Claude what this skill is and when to load it:

---
name: my-design-system
description: Reference the design system for brand guidelines, component
  patterns, colors, typography, spacing, and design tokens
allowed-tools: Read, WebFetch
---

# Your Design System

## Design Philosophy
...

After the frontmatter, write your design system in standard Markdown. Use headings for each section, code blocks for CSS, and tables for quick-reference data.

Team sync via GitHub

Store the skill in a shared GitHub repo. Team members clone it into their ~/.claude/skills/ directory. When you push an update, everyone gets the latest rules on their next git pull.

07

What to include vs. what not to include

Scope matters. Too broad and the file becomes noise. Too narrow and it doesn't catch enough mistakes.

Include
  • Patterns you've corrected twice or more
  • Font rules with exact CSS values
  • Color values and explicit "wrong" values
  • Component CSS Claude can copy directly
  • Rules with ALWAYS / NEVER language
  • Image aspect ratios by context
  • Common mistakes with corrections
Do not include
  • One-off styles for a single page
  • Temporary experiments
  • Contradictory rules (remove the old one first)
  • Overly specific rules ("the About page hero should be 500px tall")
  • Full page templates (those go in a separate skill)
  • Business logic or content strategy

The test: If you've corrected the same thing twice, it belongs in the file. If it only applies to one page, it doesn't.

08

Writing rules Claude actually follows

Claude responds to definitive language. Vague rules get vague compliance. Precise rules get precise output.

The ALWAYS / NEVER pattern

The strongest rules in any design system use ALWAYS and NEVER (capitalized). This eliminates ambiguity:

Vague

"Body text should generally be dark."

Precise

"Body text MUST ALWAYS be #000. NEVER use #555, #666, or #333 for body text."

Vague

"Use clean, modern fonts."

Precise

"Headlines: Canela Light 300, 38px desktop / 26px mobile, sentence case, 0.5px letter-spacing. NEVER use Geogrotesque for headlines."

The "common mistakes" pattern

Document what goes wrong, not just what's right. JRB's design system includes common mistakes for every major component:

Real example: JRB's Do's and Don'ts section 
### Don'ts
- Don't use hard/brutalist shadows (no 4px 4px 0px)
- Don't use 0px border-radius on interactive elements (minimum 2px)
- Don't use fully rounded corners (no 9999px)
- Don't apply grayscale filters to product images
- Don't use skew or rotate transforms for layout elements
- Don't use bright saturated colors beyond --primary-color
- Don't use viewport-unit typography (no vw sizing)
- Don't use #C73A3A or other reds as the accent
  — the brand accent is #ff2e92
- Don't use #FAF8F5 for backgrounds
  — the correct warm background is #f6f2eb
- Don't use #1A1A1A as primary black — use #000
- Don't use emojis — never use emoji characters
  anywhere in pages. Use CSS-drawn icons or inline SVGs instead.

Each rule is a lesson learned. Someone noticed Claude doing the wrong thing, and they encoded the correction so it never happens again.

Specificity of values

Concrete values always beat descriptions:

  • 16px is better than "standard size"
  • cubic-bezier(.39, .575, .565, 1) is better than "smooth easing"
  • 0 1px 2px rgba(16, 24, 40, .04) is better than "subtle shadow"
09

The feedback loop

The design system isn't a static document. It improves every time you use it. This 8-step cycle is how it grows from a basic file to a comprehensive system.

Build → Grade 1–100 → Claude explains gaps → Fix → Spot patterns → Update system → Push to GitHub → Every future build benefits
  1. Build something. Ask Claude to build a page or section.
  2. Grade it 1–100. Tell Claude: "Rate this page against our design system, 1–100."
  3. Claude grades itself. It lists what's right and what breaks rules — "78/100 — headline uses Geogrotesque instead of Canela, body text is #333 instead of #000, button is pink but should be black."
  4. Fix the issues. Tell Claude to fix the specific problems it identified.
  5. Identify patterns. If you notice the same issue coming up repeatedly, that's a signal.
  6. Update the design system. Tell Claude: "Add a rule to our design system that [new rule]."
  7. Push to GitHub. "Push the design system changes to GitHub."
  8. Every future build benefits. The new rule is now enforced automatically.
Real example: Adding a color rule

You notice Claude keeps using #555 gray for body text instead of #000 black. You say:

"I've noticed we keep getting body text in #555 gray instead of #000 black.
Add a rule to the design system that body text must always be #000."

Claude adds to the file:

### Body Text Color
- Body/description text color: ALWAYS #000
- NEVER use #555, #666, #333, or other grays for body text
- Only exception: secondary helper text (timestamps, metadata)

That rule now applies to every page Claude builds, forever.

The design system is the product. The pages are the output. Every time you catch a pattern and add a rule, every future page benefits.

10

Getting started: zero to V1

You don't need 1,500 lines on day one. A useful V1 takes 30–60 minutes. Here's the checklist.

  1. Create the file 
    mkdir -p ~/.claude/skills/my-design-system
touch ~/.claude/skills/my-design-system/SKILL.md
  2. Add the frontmatter

    Name, description, and allowed-tools (usually Read, WebFetch). See the file structure section above for the template.

  3. Start with typography

    Document your font families, sizes, weights, and usage rules. This prevents the most visible inconsistencies. Include CSS custom properties and @font-face declarations if you use custom fonts.

  4. Add your colors

    List exact hex values with CSS variable names. Include 2–3 "wrong" values to avoid — the near-matches Claude might generate.

  5. Document your button

    One component, fully specified with CSS, gives Claude a template for how detailed other components should be.

  6. Write 5 ALWAYS / NEVER rules

    The clearest, most impactful rules you already know. What's the thing you'd correct every time? Put that in the file.

  7. Build a page and run the loop

    Ask Claude to build a page. Rate it. Fix the issues. Add any new rules you discover. Your V1 will grow organically from here.

  8. Optional: Extract from Figma

    If you have Figma designs, the Figma MCP's create_design_system_rules tool can extract design tokens and generate a starting template automatically.

11

Advanced: The skill ecosystem

A design system is just one skill. As your setup matures, it becomes part of a family of interconnected skills. This is optional for V1 — but useful to see where it can go.

Here's how JRB's skills work together:

Skill Purpose Lines
Design System Tokens, rules, components — the source of truth 1,586
Section Library Catalog of 108 reusable sections with status and usage notes 200
CSS Standards Deep CSS methodology reference (BEM, specificity, spacing tokens) 486
QA Checker Post-build audit that validates against the design system —
Page Builder Complete page generation workflow that loads design system first —

The page builder loads the design system before writing any code. The QA checker validates the output against the same rules. The section library prevents duplicate work. Each skill references the others — but the design system is the foundation everything else builds on.

Start with one file. You can always split it into a family of skills later. The single SKILL.md is the foundation — everything else is an extension.
