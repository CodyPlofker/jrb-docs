1. Status — what's live right now

Three versions are deployed in parallel — V1 is the customer-facing production app, V2 was the first calibration round, V3 is the latest with Foundation Stick + inline feedback.

2. Phase history

The project moved through four named phases plus a calibration patch. Each version is a separate Vercel project, so older versions stay reachable for comparison.

MVP Late March 2026

Camera + Claude Vision proof-of-concept

Bare-bones flow: take a selfie, send to Claude Sonnet 4 Vision, get back a JSON shade recommendation. No training data, no feedback collection.

V1 Late March / Early April

Customer-shippable production app

  • UX hardening (square camera, prominent CTA, real JRB logo)
  • Real complexion shade names from CX data baked into system prompt (12 months of Richpanel conversations)
  • Tint-first results layout — Miracle Balm shown before complexion routine
  • Feedback collection added (separate /feedback page); persistent storage via Upstash Redis
V2 Early April

Seven data-driven recommendations from full training dataset

Built on top of 33,482 Richpanel shade conversations and 2,735 Junip review cross-references. Improved undertone detection rules, expanded shade vocabulary, refined fallback logic. Used for CAB (Customer Advisory Board) testing — separate /feedback page made it easy to send a focused testing link.

V3 April 11

Foundation Stick + inline feedback + calibration from CAB

  • Foundation Stick integration — new product added to results. Lookup table maps 8 skin tones × 3 undertones to confirmed shade names. Step 1 in results now shows a 3-tier coverage selector: Sheer (JETM) / Light-Medium (WTF) / Full (Foundation Stick).
  • Inline feedback — feedback form moved from a separate page to directly below the results. Higher capture rate.
  • Calibration from 89 CAB submissions: lean darker (model was skewing 1 shade too light), expand neutral bucket (default to Neutral when ambiguous), add explicit FP range guidance for Med-Dark/Dark.
V3.1 April 12

Dark/Deep FP corrections + Octane AI few-shot examples

Patch on top of V3: tightened FP shade ranges for Dark/Deep skin tones (was still recommending too light), and added few-shot examples from Octane AI conversations to the system prompt.

Headline finding

V3.1 was supposed to fix the "skewing too light" pattern, but 35% of V3 submissions still mark the result as "Too Light" — same direction of error as before V3 calibration. The "lean darker" instruction in the system prompt isn't strong enough, OR the few-shot examples aren't covering enough cases.

Secondary finding

The biggest undertone miss is Neutral (18% of submissions) — model is still picking Cool or Warm when the customer says they're Neutral. The "expand the neutral bucket" V3 instruction worked partially but needs more weight.

5. Feedback tool

How customer feedback gets from the screen into a dashboard we can act on.

Flow

  1. Customer submits selfie on V3 → sees recommendation
  2. Inline form below results asks: was the skin tone right? was the undertone right? what did you actually buy? notes?
  3. POST /api/feedback writes to Upstash Redis (key shade-feedback-entries) tagged with version: "v3"
  4. Dashboard at shade-feedback-dashboard/index.html fetches via GET /api/feedback, renders password-gated table with filters by status

What's captured

6. Punch list — what to do next

Ordered by what would move the needle most.

1
P1

Fix the "too light" pattern

35% of V3 submissions still mark the result "Too Light" despite V3.1 calibration. Two possible angles: (a) strengthen the "lean darker" instruction in the V3 system prompt with stronger language and concrete examples; (b) audit which skin-tone buckets are most often wrong (pull from Redis, group by skinToneDetected) and patch the few-shot examples for those specific buckets. Source file: src/app/api/match/route.ts.

2
P1

Confirm Foundation Stick shade mapping with product team

The 8×3 lookup table in src/lib/shade-data.ts was built from quiz-logic.html and is marked provisional. Get Cheyenne or product to sign off before promoting V3 to production. The table is in the technical appendix below.

3
P2

Expand the neutral bucket further

Second-largest miss (18% of undertone errors are "should be Neutral"). The V3 instruction said "default to Neutral when ambiguous" — clearly not strong enough. Try: invert the rule. Require positive evidence to assign Cool or Warm; default to Neutral otherwise.

4
P2

Pull more dark-tone Richpanel conversations

Re-run shade-matching-data/extract-cc-conversations.py (it timed out mid-run; progress JSON is at cc-extraction-progress.json). Goal: get the dark-tone CX training corpus from "thin" to "comparable to light/medium." Then add new few-shot examples for Med-Dark / Dark / Deep buckets.

5
P2

Share V3 link internally

Drop jrb-shade-ai-v3.vercel.app in #ai-shadematching (Slack channel C091ZCS2CR3) so we get more feedback volume. Currently averaging ~5 submissions/day — would benefit from a wider net.

6
P3

Promote V3 → production

Once accuracy is past ~70% on skin tone (currently 42%) and the Foundation Stick mapping is confirmed, deploy V3 code to the V1 Vercel project so customers see it on the production URL. Deploy recipe is in the technical appendix.
